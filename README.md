# 洞察 · 淬炼 · 验证（insight-refine-verify）

> 一套结构化的代码改动方法论。**先想清楚再动手，动手后反复检查，确认无误再上线。**
>
> 适用于：bug 修复 · 功能添加 · 代码重构 · 安全审查 · 架构变更

---

## 这是什么？

你有没有遇到过以下情况？

- 🐛 **"修了一个 bug，又引出两个新 bug"** — 改之前没多想，改完发现影响了别的模块
- 🕐 **"这个简单，10分钟改完" — 结果改了一下午"** — 低估了复杂度，边想边改边出问题
- 📝 **"我改了什么来着？"** — 改动太多，提交时已经说不清改了什么、为什么改
- 🚀 **"上线了应该没问题" — 然后出问题了"** — 没有测试或测试不充分，全靠信心上线

**这套方法论就是为了解决这些问题。** 它不是一套死板的规定，而是一个框架，告诉你"改代码之前想什么、改的时候检查什么、改完之后验证什么"。

---

## 三条铁律

> 任何一条违反了，这次改动的质量就已经失控。

| # | 铁律 | 反面例子 |
| --- | --- | --- |
| 1 | **没找到根因，绝不改代码** | "试试加个 sleep 看看会不会好" |
| 2 | **一次只解决一个问题** | "既然改到这里了，顺手把那个也修一下吧" |
| 3 | **产出必审核，审核必记录** | "我觉得没问题，直接上" |

## 快速开始（30 秒安装到 Trae IDE）

### 方式一：项目级安装（只在当前项目生效）

在 Trae IDE 的终端里执行：

```bash
# Windows PowerShell
mkdir -p .trae/skills/insight-refine-verify
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/maze9929/insight-refine-verify/main/SKILL.md" -OutFile ".trae/skills/insight-refine-verify/SKILL.md"
```

文件结构应该是：
```
你的项目/
└── .trae/
    └── skills/
        └── insight-refine-verify/
            └── SKILL.md   ← 就是这一步放进去的
```

### 方式二：全局安装（所有项目自动生效）

```bash
# Windows PowerShell
mkdir -p "$env:USERPROFILE\.trae-cn\skills\insight-refine-verify"
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/maze9929/insight-refine-verify/main/SKILL.md" -OutFile "$env:USERPROFILE\.trae-cn\skills\insight-refine-verify\SKILL.md"
```

### 方式三：手动安装（最简单）

1. 打开 [SKILL.md](https://github.com/maze9929/insight-refine-verify/blob/main/SKILL.md)
2. 复制全部内容
3. 在你的项目里创建文件 `.trae/skills/insight-refine-verify/SKILL.md`
4. 粘贴进去保存

**完成之后就可以用了！**

## 怎么使用

### 在 Trae IDE 中

安装好 Skill 之后，不需要做任何特别的事情。当你说类似的话时，AI 会自动识别并加载本方法论：

| 你说的话 | 触发说明 |
| --- | --- |
| "帮我修一个 bug，接口返回数据不对" | 自动按 Phase 1-5 流程执行 |
| "给这段代码加个功能" | 走完整流程，确保改动可控 |
| "重构一下这个模块，让它更清晰" | 不盲目动手，先分析再改 |
| "检查一下这里有没有安全问题" | 安全审查流程 |

### 典型对话示例

**场景 1：修一个接口返回值错误的 bug**

```
你：修个 bug，/api/users 接口返回了重复数据。

AI 按流程执行：
├── Phase 1 洞察：先不写代码
│   ├── 问题登记卡：现象/复现步骤/期望/实际
│   ├── 定位到具体代码行（对半分割）
│   └── 5 Whys 分析根因（不是"变量写错了"，而是"缺少唯一性校验"）
│
├── Phase 2 淬炼：写方案 → 审核 → 写代码 → 再审核
│   ├── Before/After 对比
│   ├── 作者自检 8 项（逻辑/异常/边界/安全/性能...）
│   └── 代码三轮审核
│
├── Phase 3-4：影响分析 + 测试验证
│
└── Phase 5：结案报告
```

**场景 2：添加一个新功能**

```
你：加一个"用户收藏文章"的功能。

AI 按流程执行：
├── Phase 1：现有代码结构分析（用户模块、文章模块、数据库 schema）
├── Phase 2：方案设计（新接口 POST /api/favorites + 数据库表 favorites）→ 方案审核 → 实现 → 自检 → 代码审核
├── Phase 3：对现有用户接口和文章接口的影响分析
├── Phase 4：测试（正常/异常/边界）
└── Phase 5：结案
```

### 手动指定强度

你可以明确告诉 AI 用哪个强度：

- **"简单改一下，L0"** → 只改不改瞎想，快速处理
- **"按 L2 标准流程来"** → 正常复杂度的改动
- **"这里很重要，走 L3"** → 高风险改动，完整流程

## 完整流程一览

```
Step 0 选强度（0-18分 → L0~L4）
    │
    ▼
Phase 1 洞察（只读不改）
    ├── 问题登记卡
    ├── 调用链定位（对半分割）
    └── 5 Whys 根因分析
    │
    ▼
Phase 2 淬炼
    ├── 写方案文档（Before/After + 不改动范围 + 替代方案 + 回滚）
    ├── 方案审核
    ├── 最小化实现（≤200行，不捎带，不重构）
    ├── 作者自检 8 项
    └── 代码三轮审核
    │
    ▼
Phase 3 分析（L2 起做）
    ├── 影响分析矩阵
    └── 风险分级 + 回滚预案
    │
    ▼
Phase 4 验证
    ├── 可回滚部署
    ├── 测试覆盖（正常→异常→边界→回归→并发）
    └── 日志观测（确认代码走了预期路径）
    │
    ▼
Phase 5 闭环
    ├── 失败 → 回 Phase 1 重来
    └── 通过 → 写结案报告
```

### 五个强度级别

| 级别 | 总分范围 | 适用场景 | 做什么 | 预计耗时 |
| --- | --- | --- | --- | --- |
| **L0** | 0-2 分 | 简单 typo、错别字、改一行 | 改完肉眼确认 | 5-15 分钟 |
| **L1** | 3-6 分 | 小 bug、简单修改、加个工具函数 | 登记卡 + 实现 + 简单测试 | 15-45 分钟 |
| **L2** | 7-10 分 | 普通 bug、中等功能、重构一个模块 | 标准五阶段 | 45-120 分钟 |
| **L3** | 11-14 分 | 复杂 bug、新功能、多个模块改动 | 完整流程 + 影响分析 + 完整测试 | 2-4 小时 |
| **L4** | 15-18 分 | 架构变更、核心模块重写、涉及安全/支付 | 完整流程 + 灰度发布 + 指标监控 | 4 小时+ |

### 强制升级规则（跳级触发）

只要满足以下任意一条，不管打分多少，必须升级：

| 触发条件 | 升级到 |
| --- | --- |
| 同一个问题出现 ≥ 2 次 | **L3** |
| 改动涉及 ≥ 3 个文件/模块 | **L2** |
| 改动删除了现有代码（非仅新增） | **L2** |
| 改变了函数签名 / API 接口定义 | **L3** |
| 涉及安全 / 权限 / 加密 / 支付 | **L3** |

```

【三条铁律】

2. 一次只解决一个问题
3. 产出必审核，审核必记录

Phase 5: 闭环（结案报告 / 重来）

详细规则请参考：https://github.com/maze9929/insight-refine-verify
```

## 常见问题 FAQ

**Q: 会不会太慢？每个改动都走完整流程岂不是很花时间？**

A: 不会。L0 和 L1 都很快（5-45 分钟），只有真正复杂的改动才走 L2/L3。
**花时间的地方不是流程，花时间的是思考。** 省略流程等于省略思考，结果就是反复返工反而更慢。

**Q: 我就是不想按流程来，我想直接写代码怎么办？**

A: 没问题！L0 级别就是给你"快速改一下"用的。但请你在提交前至少做一遍"作者自检 8 项"。

**Q: 这套方法论和 TDD（测试驱动开发）是什么关系？**

A: 互补。TDD 告诉你"先写测试再写代码"，这套方法论告诉你"在写测试之前，你应该先想清楚根因是什么、改动范围是什么、会影响谁"。

**Q: 单人开发也需要吗？**

A: 更需要。团队开发有人帮你 review，单人开发只有你自己。方法论就是你的"虚拟 reviewer"。
