# 洞察 · 淬炼 · 验证

> **三条铁律**：① 没找到根因，绝不改代码 ② 一次只解决一个问题 ③ 产出必审核，审核必记录
>
> **详细模板和案例**：参见项目根目录 `方法论-洞察淬炼验证-v1.1.md`

---

## Step 0：选强度

**打分**（每维 0-3 分）：严重级别 / 改动量 / 影响面 / 首次出现 / 时间压力 / 并发状态

**总分 → 强度**：0-2 → L0 随手修 | 3-6 → L1 简化 | 7-10 → L2 标准 | 11-14 → L3 完整 | 15-18 → L4 重大

**强制升级**（触发任一条直接跳级）：
- 同一问题 ≥ 2 次 → L3
- 改动涉及 ≥ 3 个文件 → L2
- 删除了现有代码 → L2
- 改变了函数签名/接口 → L3
- 涉及安全/权限/加密/支付 → L3

---

## Phase 1 · 洞察（只读不改）

**做什么**：
1. 填问题登记卡：现象 / 复现步骤 / 期望 / 实际 / 影响 / 级别 / 已复现？（L1: ≤5行）
2. 画调用链 + 对半分割：画路径 → 中点检查 → 排除一半 → 递归到单行
3. 5 Whys 根因分析：追问 3-5 次"为什么"，挖到系统级原因（L2: ≥3次，L3+: 5次）

**检查什么**：
- [ ] 登记卡完整
- [ ] 可稳定复现
- [ ] 已定位到具体代码位置
- [ ] 根因明确（系统级，不是"某人粗心"）
- [ ] **没有写任何修改代码**

**输出**：问题登记卡 + 调用链图 + 根因结论

---

## Phase 2 · 淬炼

**做什么**：
1. 写方案文档：改动点 / Before→After / 不改动范围 / 兼容性 / 替代方案(≥1个) / 回滚方案 / 预计影响
2. 方案审核：实现者视角(L2) → 维护者视角(L3) → 系统视角(L3)
3. 最小化实现：≤200行，不捎带其他问题，不重构，YAGNI
4. 作者自检 8 项：逻辑 / 异常 / 边界 / 安全 / 性能 / 可读 / 一致 / 日志
5. 代码审核：逻辑正确性(L1+) → 回归测试(L2+) → 全局搜索同类(L3+)

**检查什么**（自检 8 项黄金比例）：
- 正确性 40%：if/else 全考虑？空值/null 处理？
- 可读性 25%：变量名语义化？复杂逻辑有注释？
- 性能 15%：循环里有 DB？O(n²)？
- 安全 12%：输入验证？注入风险？
- 其他 8%

**输出**：方案文档 + 已审核代码

---

## Phase 3 · 分析（L2 起）

**做什么**：
1. 填影响矩阵：核心/边缘/DB/API/缓存/日志/错误路径（L2: 2-3模块，L3+: 全7项）
2. 风险分级 P0-P3 + 回滚预案（L2: 只回答"5分钟能回滚吗？"）

**检查什么**：
- 净影响为 `-` 的项必须有处理策略
- P0/P1 必须有明确回滚步骤

**输出**：影响矩阵 + 风险分级 + 回滚方案

---

## Phase 4 · 验证

**做什么**：
1. 可回滚部署：备份旧文件 / 验证部署命令 / 回滚命令就绪 / 低峰期(P0-P1)
2. 测试覆盖：正常+异常(L1+) → +边界+回归(L2+) → +并发+历史失败(L3+)
3. 日志观测：触发日志+无新ERROR(L2+) → +性能+异常路径(L3+)

**检查什么**：
- [ ] 旧文件已备份
- [ ] 回滚命令可一键执行
- [ ] 测试 100% PASS
- [ ] 日志中确认代码走了预期路径

**输出**：测试报告 + 日志截图

---

## Phase 5 · 闭环

**做什么**：
- 测试失败 → 回 Phase 1 重来（不要在当前流程上打补丁）
- 测试通过 → 写结案报告

**结案报告**：问题简述 / 根因(引用5 Whys) / 改动摘要 / 测试结果 / 风险评估 / 防复发措施(5 Whys→规范/清单/自动化) / 耗时
（L2: 3行，L3+: 完整）

**L3+ 额外**：方法论自我改进 5 问（最花时间？走过场？漏检？引入新bug？重做会多花时间在哪？）

---

## 升级 / 降级

**升级**：复杂度 ≥ 估计 3 倍 / 审核 ≥ 2 个独立问题 / 测试失败 > 20% / 触发强制规则

**降级**：5 分钟定位+根因极简 / ≤10 行+0 问题 / 100% PASS（最低 L0）
