# 📘 「洞察 · 淬炼 · 验证」方法论 · v1.1

> **一句话说清它是什么**：你在做一个改动（改代码 / 改方案 / 改文档）之前、之中、之后，应该按什么顺序想哪些事情。这套方法论告诉你"先想什么，后想什么，哪些步骤可以跳过"。
> **适用**：Bug 修复 / 功能添加 / 代码重构 / 性能优化 / 安全审查 / 架构变更 / 文档审核 / 方案评估
> **不适用**：纯实验性探索（MVP）/ 一次性脚本

---

## 📍 入口导航：读文档之前，先花 1 分钟判断你该走多深

> **分级系统的唯一目的**：告诉你哪些步骤要认真做，哪些可以简化或跳过。它本身不是重点，它是一张"地图索引"。

### 快速打分

| 维度 | 0 分（最轻） | 1 分 | 2 分 | 3 分（最重） |
| --- | --- | --- | --- | --- |
| 严重级别 | 纯代码风格/typo | 边缘功能 bug | 重要功能异常 | 系统不可用 / 数据丢失 |
| 改动量 | ≤ 1 行 / 1 文件 | ≤ 10 行 / 1-2 文件 | 10-100 行 / 2-5 文件 | > 100 行 / ≥ 5 文件 |
| 影响面 | 只有你自己 | 团队内部工具 | 内部核心系统 | 客户可直接感知 |
| 是否首次出现 | 已知问题 / 之前修过 | 第一次出现，影响小 | 第一次出现，影响中 | 反复出现 ≥ 2 次 |
| 时间压力 | 无 | 本周内 | 今天必须 | 分钟级 |
| 并发/状态 | 不涉及 | 简单状态 | 复杂状态机 | 多节点/分布式 |

**总分 = ________** → 决定走哪个强度：

| 总分 | 强度 | 对应流程深度 |
| --- | --- | --- |
| 0-2 | **L0 随手修** | 只看 Phase 2 的"最小化实现" + Phase 4 的"肉眼确认" |
| 3-6 | **L1 简化流程** | Phase 1（简要）→ Phase 2（跳过方案文档）→ Phase 4 |
| 7-10 | **L2 标准流程** | Phase 1-5 全走，每阶段按标注的"标准流程"深度执行 |
| 11-14 | **L3 完整流程** | Phase 1-5 全走，每阶段按"完整流程"深度执行 |
| 15-18 | **L4 重大变更** | 全流程 + 第二人独立审核 + 灰度发布 |

### 强制升级规则（不能商量）

```
① 同一问题 ≥ 2 次 → 至少 L3
② 改动涉及 ≥ 3 个文件/模块 → 至少 L2
③ 改动删除了现有代码（非仅新增）→ 至少 L2
④ 改变了函数签名/接口约定 → 至少 L3
⑤ 涉及安全/权限/加密/支付 → 至少 L3
```

### 懒得打分？用快速查表

| 一句话描述 | 建议强度 |
| --- | --- |
| "就是个 typo" | L0 |
| "加个空值检查，不然会崩" | L1 |
| "这个接口好像返回的数据不对" | L2 |
| "同样的 bug 第三次出现了" | L3（触发规则①） |
| "用户登录有时候会登不上" | L2 或 L3 |
| "支付功能偶尔扣款失败" | L4（触发规则⑤） |
| "重构这段逻辑，更清晰" | L2（触发规则③） |
| "把一个函数拆成两个，方便复用" | L1 或 L2 |

# 🚀 方法论正文：各阶段详解

# Phase 1 · 洞察：理解问题，定位根源

> **核心原则**：只读不改。这一阶段写的每一个字，都是为了搞清楚"到底发生了什么"，不是为了开始修复。
> **各强度要求**：L0 跳过 / L1 简要登记卡 + 快速定位 / L2 登记卡 + 调用链 + 至少 3 Whys / L3-L4 全套

## 1-1 问题登记卡：让问题"跑不掉"

**填写下面这张卡，不写完整不进入下一步。**

```
📋 问题登记卡
├── 现象描述：_______________________________________
│   （用户看到了什么？系统做了什么不应该做的事？
│    用具体的错误信息、状态值、可见行为来描述，不要写推测）
│
├── 复现步骤：_______________________________________
│   1. _______________________________________________
│   2. _______________________________________________
│   3. _______________________________________________
│   ...（必须精确到点击按钮/输入数据/访问路径）
│
├── 期望结果：_______________________________________
│   （应该发生什么？精确到具体的输出文本或状态值）
│
├── 实际结果：_______________________________________
│   （实际发生了什么？精确到错误信息、状态码、日志行号）
│
├── 影响范围：_______________________________________
│   （哪些功能受影响？影响多少用户？）
│
├── 严重级别：□ P0 阻断  □ P1 高  □ P2 中  □ P3 低
│
└── 已复现？：□ 是（每次 / 偶发）  □ 否（还在复现中）
```

**💡 提示**：如果"复现步骤"这一栏你写不出来，那就停在这里 —— 不能复现的问题不能修。先想办法稳定复现。

**L1 简化版**：≤ 5 行即可（现象 + 预期 + 实际 + 严重级别）。

## 1-2 调用链 + 对半分割：系统化定位

调试不是从头到尾读代码，调试是**二分查找**。

**步骤 A：画出从输入到输出的完整路径**

```
用户操作 → 入口函数/路由 → 业务逻辑层 → 数据访问层 → 外部依赖（DB/缓存/第三方API）→ 返回结果
```

**步骤 B：在路径中点插入"检查点"**

取整条路径的中点，检查那里的状态是否正确。例如路径是"请求 → 路由 → Service → 数据库"，就在 Service 层检查返回值。

**步骤 C：根据检查结果消除一半代码**

- ✅ 中点状态正确 → bug 在**后半段**（Service 到返回结果之间）
- ❌ 中点状态错误 → bug 在**前半段**（请求到 Service 之间）
- 递归重复，直到定位到**单个函数 / 单行代码**

**💡 实战案例**：

> **现象**：用户登录后页面显示"欢迎 undefined"。
>
> **路径**：点击登录 → POST /api/login → `authService.login()` → `userRepository.findByEmail()` → DB 查询 → 返回 user 对象 → 前端渲染
>
> **中点检查**（Service 层）：`authService.login()` 返回 `{ id: 42, name: null, email: "alice@example.com" }` → name 为 null，bug 在前半段（Service 及之前）
>
> **再二分**：检查 `userRepository.findByEmail()` → 返回 `{ id: 42, name: null, email: "..." }` → name 字段就是 null，说明 bug 在 Repository 层
>
> **定位**：检查 SQL 查询 → 发现 SELECT 语句遗漏了 `name` 列
>
> **结论**：根因是数据库查询语句缺少字段。3 次检查就定位到了，不需要从头到尾读 1000 行代码。

## 1-3 5 Whys 根因分析：从代码挖到系统

找到了代码层面的 bug 不算完。**必须追问至少 3-5 次"为什么"**，才能从"修一个 bug"升级到"防止一类 bug 再次发生"。

**模板**：

```
🔍 5 Whys 分析
├── 现象：_________________________________________
│
├── Why 1：为什么会发生这个现象？
│   答：___________________________________________
│
├── Why 2：为什么 Why-1 的原因会存在？
│   答：___________________________________________
│
├── Why 3：为什么 Why-2 的原因会存在？
│   答：___________________________________________
│
├── Why 4：为什么 Why-3 的原因会存在？
│   答：___________________________________________
│
├── Why 5：为什么 Why-4 的原因会存在？
│   答：___________________________________________
│
└── 根因结论：_____________________________________
    （通常是流程/规范/架构层面的问题，不是"某人粗心"）
```

**💡 实战案例**：

> **现象**：特殊字符的密码导致登录失败。
>
> Why 1：为什么登录失败？→ 数据库查询时特殊字符被解释为 SQL 语法，查询失败。
> Why 2：为什么特殊字符没被处理？→ 没有使用参数化查询，直接字符串拼接 SQL。
> Why 3：为什么没有使用参数化查询？→ 这个模块是早期"快速验证"写的，当时没考虑安全问题。
> Why 4：为什么这种代码能合并到主分支？→ 代码审查时只看功能是否正常，没有检查安全性。
> Why 5：为什么审查时没检查安全性？→ 团队没有统一的安全审查清单，全靠个人经验。
>
> **根因结论**：缺少标准化的安全审查清单，导致不安全的数据库查询方式进入了代码库。
>
> **修复策略扩展**：不仅要把这次的字符串拼接改为参数化查询，还要在代码审查清单中加入"SQL 注入风险检查"项。→ **这就是 5 Whys 的系统级结论直接导向防复发措施**。

## ✅ Phase 1 完成检查清单

在进入 Phase 2 之前，必须满足以下全部条件：

- [ ] 问题登记卡填写完整
- [ ] 问题可稳定复现（或偶发但已明确触发条件）
- [ ] 通过调用链 + 对半分割法定位到具体的代码位置
- [ ] 5 Whys 分析完成，明确了系统级根因（L2 至少 3 Whys，L3+ 必须 5 Whys）
- [ ] **没有写任何修改代码**

# Phase 2 · 淬炼：设计方案，写出经过审核的代码

> **核心原则**：先写方案再写代码。方案是你对问题的理解产物，代码是方案的执行产物。
> **各强度要求**：L0 直接改 + 肉眼确认 / L1 跳过方案文档、但必须做自检 / L2 写方案 + 一轮审核 + 自检 + 两轮代码审核 / L3-L4 全套

## 2-1 方案文档：写清楚"改什么"再动手

**模板**：

```
📐 改动方案
├── 改动点：_______________________________________
│   改几个地方？哪些文件？哪些函数？
│
├── Before → After：________________________________
│   修改前的逻辑（或引用代码片段）
│   修改后的逻辑
│
├── 不改动范围：___________________________________
│   明确写清楚"不动什么"——证明你思考过边界
│
├── 兼容性：□ 完全向后兼容  □ 有条件兼容  □ 破坏性变更
│
├── 替代方案（至少 1 个）：___________________________
│   "如果不这么改，还能怎么改？为什么选当前方案？"
│   注意：至少有一个替代方案应该是"完全不同的思路"
│   （比如完全换一种架构，而不是小改动）
│
├── 回滚方案：_____________________________________
│   "上线后如果出问题，怎么在 5 分钟内回到旧版本？"
│
└── 预计影响：_____________________________________
    （性能影响 / 代码行数变化 / 新增依赖）
```

## 2-2 方案审核：从三个角度拷问你的方案

| 轮次 | 身份 | 核心问题 | 判断标准 |
| --- | --- | --- | --- |
| 第一轮 | **实现者** | "我能按这个方案写出来吗？" | 每一步都具体到可执行吗？有没有遗漏的细节？伪代码能跑通吗？ |
| 第二轮 | **维护者** | "三个月后我回来能看懂吗？" | 逻辑直观吗？变量名语义化吗？出问题能从日志定位到这里吗？ |
| 第三轮 | **系统整体** | "会牵一发而动全身吗？" | 搜索所有调用方：它们的行为会变吗？有没有依赖这个接口的其他模块？ |

**L2 及以下**：可以合并为一轮审核（重点看实现者视角）。
**L3 及以上**：必须三轮全过。

## 2-3 最小化实现：一个问题 = 一次改动

**执行原则**：

- **不顺便修其他问题**：哪怕旁边那行代码看着也不对，也必须记下来单独处理
- **不重构**：重构是单独的任务，和 bug 修复混在一起 = 两件事都做不好
- **不追求"未来可扩展性"**：YAGNI 原则（You Aren't Gonna Need It）— 如果目前不需要，就不为"将来可能需要"写代码
- **体量控制**：每次修改 ≤ 200 行。超过了说明一次做了太多事情，需要拆分

## 2-4 作者自检 8 项：写完代码后，你是第一道防线

> 在发给别人 review 之前，先过这张清单。逐项检查，而不是扫一眼。

| # | 检查项 | 通过标准 |
| --- | --- | --- |
| 1 | **逻辑正确性** | 每个 if/else 分支都考虑过了吗？空值/零值/空列表/null 都处理了吗？ |
| 2 | **异常处理** | try/catch 覆盖了所有可能失败的调用吗？错误信息对调试有帮助吗？ |
| 3 | **边界值** | 输入为 0、最大整数、空字符串、超长字符串、特殊字符时行为可预测吗？ |
| 4 | **安全性** | 用户输入经过验证了吗？没有 SQL 注入 / XSS / 路径遍历风险吗？ |
| 5 | **性能** | 循环里有没有数据库查询？有没有 O(n²) 的操作？ |
| 6 | **可读性** | 变量名/函数名能表达意图吗？复杂逻辑有解释性注释吗？ |
| 7 | **一致性** | 代码风格和项目中已有代码一致吗？命名惯例一致吗？ |
| 8 | **日志** | 关键路径有日志吗？异常有 ERROR 级别日志吗？日志中能看到关键参数吗？ |

**审查时间分配建议**（黄金比例）：

```
功能正确性 40% ────────────────────
代码可读性 25% ──────────────
性能考量   15% ─────────
安全因素   12% ──────────
其他        8% ──────
```

> 这不是硬性规定，而是提醒：**大部分审查时间应该花在"代码逻辑对不对"上，而不是"变量名好不好看"上。**

## 2-5 代码三轮审核

| 轮次 | 重点 | 具体做什么 |
| --- | --- | --- |
| 第一轮 | **逻辑正确性** | 手动模拟执行一遍。用几个典型输入（包括边界值），在脑子里走一遍代码的每一行。每一步的结果和你预期一致吗？ |
| 第二轮 | **不破坏已有功能（回归）** | 全局搜索所有调用被修改函数的地方。确认这些调用方的行为不受影响，或影响在预期之内。跑一遍已有的测试套件。 |
| 第三轮 | **全局搜索对比** | 搜索代码库中类似的模式（类似的空值处理、同样的字符串拼接、相同的 API 调用方式）。确认其他代码中有没有同样的问题。 |

**L1**：只做第一轮。
**L2**：做前两轮。
**L3-L4**：三轮全做，且第三轮之后如果发现了同类问题但不在本次改动范围内，需要记录为"后续任务"。

# Phase 3 · 分析：看清改动对全局的影响

> **核心原则**：在部署前，明确这个改动对整个系统意味着什么。不是"改完就完事了"。
> **各强度要求**：L0-L1 跳过 / L2 简单影响分析 + 回答"出问题怎么回滚" / L3 完整 7 项影响矩阵 + P0-P3 风险分级 / L4 + 灰度发布计划

## 3-1 影响分析矩阵

```
📊 影响分析矩阵
┌──────────────────┬────────────┬────────────┬──────────┐
│ 模块/功能        │ 修改前行为 │ 修改后行为 │ 净影响   │
├──────────────────┼────────────┼────────────┼──────────┤
│ 核心路径         │            │            │  0/+/-  │
│ 边缘功能         │            │            │  0/+/-  │
│ 数据库交互       │            │            │  0/+/-  │
│ 外部 API 调用    │            │            │  0/+/-  │
│ 缓存策略         │            │            │  0/+/-  │
│ 日志/监控        │            │            │  0/+/-  │
│ 错误处理路径     │            │            │  0/+/-  │
└──────────────────┴────────────┴────────────┴──────────┘
```

**净影响标注**：`0` = 无变化 / `+` = 改善 / `-` = 负面影响。负影响必须有对应的说明和处理策略。

**L2 简化版**：不需要填完整矩阵，只关注 2-3 个核心模块。

## 3-2 风险分级与回滚预案

| 风险级别 | 定义 | 部署要求 | 回滚要求 |
| --- | --- | --- | --- |
| **P0** | 可能导致系统不可用 / 数据丢失 | 独立回滚脚本；低峰期部署；有人值守 | 5 分钟内可回滚 |
| **P1** | 重要功能可能受影响 / 用户可感知 | 完整回归测试；部署后 10 分钟内有人确认 | 10 分钟内可回滚 |
| **P2** | 边缘功能 / 性能微调 | 必须有测试验证；可以异步确认 | 30 分钟内可回滚 |
| **P3** | 代码风格 / 文档 / 纯内部重构 | 基本测试即可 | 按需回滚 |

**L2 简化版**：不需要填表，只需要回答一个问题："如果上线出问题，5 分钟内能回到旧版本吗？能就写清楚怎么操作，不能就说明需要准备什么。"

# Phase 4 · 验证：真实环境中确认修复有效

> **核心原则**：测试通过 ≠ 代码正确。必须看真实运行日志。口头说"没问题"不算。
> **各强度要求**：L0 肉眼确认改前改后对比 / L1 正常路径 + 异常路径各测 1 个 / L2 正常 + 异常 + 边界 + 回归 / L3 全 5 类覆盖 + 日志观测 / L4 + 性能测试 + 改前改后指标对比

## 4-1 可回滚部署

**检查清单**：

- [ ] 旧文件/旧配置已备份（带时间戳）
- [ ] 部署命令已在测试环境验证过
- [ ] 重启命令明确，且可以一键执行
- [ ] 回滚命令写在同一份文档里，随时可用
- [ ] 部署时间安排在低峰期（P0/P1 级别改动）

## 4-2 测试用例覆盖

**每个测试用例必须遵循这个格式**：

```
🧪 测试用例
├── 编号：T1 / T2 / T3 ...
├── 场景：___________________________
├── 输入：___________________________
├── 期望输出：_______________________（精确到具体值）
├── 实际输出：_______________________（运行后填入）
└── 判定：□ PASS  □ FAIL
```

**必须覆盖的场景类型**：

| 类型 | 说明 | 例子 |
| --- | --- | --- |
| **正常路径** | 典型输入下的预期行为 | "正确密码登录应跳转首页" |
| **异常路径** | 各种错误输入的处理 | "空密码应提示'密码不能为空'" |
| **边界值** | 输入的极限情况 | "密码长度恰好为最小长度" |
| **回归测试** | 确保没破坏已有功能 | "忘记密码功能仍然能用" |
| **并发/性能** | 高并发下行为（L3 起） | "20 个并发登录不出现死锁" |
| **复现历史失败** | 反复出现的问题（L3 起，如触发强制升级规则①） | "之前导致失败的所有场景再跑一遍" |

## 4-3 运行时日志观测

> **这是最容易被跳过的步骤，也是最容易发现隐藏问题的步骤。**

部署并触发修复代码后，在日志中 grep 以下内容：

1. **触发日志**：你加的日志行出现了吗？格式正确吗？参数值合理吗？
2. **无新增 ERROR**：修改前后对比，有没有新的 ERROR 或 WARNING 出现？
3. **性能日志**：关键操作的耗时有没有显著变化？
4. **异常路径日志**：故意触发一次错误场景，看日志是否能帮助定位（不要只测 happy path）

**L2 及以下**：只检查 1 和 2。
**L3-L4**：四项全查，且需要改前改后的对比数据（如果有的话）。

# Phase 5 · 闭环：记录经验，改进方法论

> **核心原则**：不记录经验，同一个坑下次还会踩。记录不只是为了自己，也是为了团队。
> **各强度要求**：L0-L1 跳过 / L2 3 行总结 / L3 完整结案报告 / L4 完整报告 + 防复发措施落地 + 方法论改进分析

## 5-1 如果测试失败：回到 Phase 1 重来

> 发现问题时，**不要在当前流程上"打补丁"**。退回去重新分析根因，完整流程重来一遍。因为"测试失败"意味着之前的根因分析可能不准确，或修复方案有疏漏。

## 5-2 如果测试通过：结案报告

```
📝 结案报告
├── 问题简述（一句话）：____________________________
├── 根因分析：____________________________________
│   （引用 Phase 1 的 5 Whys 结论）
├── 改动摘要：____________________________________
│   （文件 + 代码行数 + 关键变化）
├── 测试结果：__ / __ PASS
├── 风险评估：P__ 级别，回滚方案已确认
├── 防复发措施：__________________________________
│   （根据 5 Whys 的系统级结论，应该加什么规范/清单/自动化检查？
│    例如：5 Whys 挖到"缺少安全审查清单"→ 防复发措施 = 在审查清单中加入 SQL 注入检查项）

```

**L2 简化版**：只填问题简述 + 根因分析 + 测试结果（3 行足够）。

## 5-3 方法论自我改进（L3 起）

# 🔄 升级与降级规则

### 升级（从轻跳到重）

| 情况 | 动作 |
| --- | --- |
| Phase 1 发现实际复杂度 ≥ 最初估计的 3 倍 | 升级一级，补做跳过的步骤 |
| 审核中发现 ≥ 2 个独立问题 | 升级一级 |
| 测试失败率 > 20% | 回到 Phase 1 重新分析，同时升级 |
| 触发任意一条强制升级规则 | 直接跳到规则指定级别 |

### 降级（从重跳到轻）

| 情况 | 动作 |
| --- | --- |
| Phase 1 5 分钟内清晰定位，根因极其简单 | 后续步骤降级一级 |
| 方案 + 代码 ≤ 10 行，审核 0 问题 | 后续步骤降级一级 |
| 测试 100% PASS，无任何意外发现 | 后续步骤降级一级 |

**最低不低于 L0**。

# ⚠️ 负面影响与缓解措施（诚实披露）

> 任何方法论都有代价。了解这些，才能正确判断什么时候用、怎么用。

| 风险 | 说明 | 严重度 | 缓解措施 |
| --- | --- | --- | --- |
| **时间开销** | L0 的 3 倍耗时可能让人觉得繁琐 | 中 | 五级强度选择器 + 每阶段时间上限 + 从轻开始允许升级 |
| **检查框文化** | 机械填表而非真思考 | 高 | 每轮审核必须"举出 1 个反例"；Phase 1 结对做；定期随机抽样复盘 |
| **分析瘫痪** | 过度分析延迟行动 | 中 | 设阶段时间上限；超时直接升级或请求第二人介入 |
| **审核边际效益递减** | 第 3 轮审核只发现 0-5% 的问题 | 低 | L2 及以下合并审核轮次；"连续两轮无问题可提前终止" |
| **单人团队盲区** | 自己审自己效果差 | 中 | 时间隔离法（写完 4 小时后再审）；角色代入法（"如果这是同事写的"） |
| **抑制创造力** | 引导局部修复而非架构级改进 | 中 | 方案文档强制包含 1 个"完全不同思路"的替代方案；同类问题 ≥ 3 次强制架构讨论 |
| **文档维护成本** | 大量中间产物 | 低 | 只保留结案报告为永久记录（≤1 页 A4） |

# 🔄 迁移到非代码场景

> 方法论的核心是**结构化思考**，不是"怎么写代码"。以下是常见场景的适配方式。

### 文档审核（方案文档、设计文档）

| 方法论步骤 | 文档版对应动作 |
| --- | --- |
| 画调用链 → 对半分割 | 画文档结构图 → 取一半位置检查逻辑是否贯通 |
| 代码逻辑正确性 | 文档内部一致性检查（同一术语含义始终相同？前后结论是否矛盾？） |
| 异常路径审核 | 反例/边界场景检查（如果文档的前提假设不成立，会怎样？） |
| 搜索调用点 | 搜索文档中每个关键概念，检查是否在全文中被一致使用 |
| 日志观测 | 文档是否有可追溯的版本记录？有没有"为什么这么决策"的说明？ |

### 方案/设计评审

| 方法论步骤 | 方案评审版对应动作 |
| --- | --- |
| 问题登记卡 | 需求/目标陈述（这个方案解决什么问题？） |
| 5 Whys 根因分析 | 为什么需要这个方案？深层需求是什么？ |
| Before/After 对比 | 现状 vs 方案实施后的状态对比 |
| 影响分析矩阵 | 对现有流程/团队/技术栈的影响分析 |
| 测试用例 | "如果方案落地，怎么验证它有效？" |

### 通用迁移原则

把"代码"替换成你当前的产出物：

- **逻辑正确性** → **论证有效性**
- **异常处理** → **反例/边界场景**
- **安全性** → **风险点/漏洞/盲点**
- **性能** → **成本/效率/可扩展性**
- **可读性** → **清晰度/可理解性**
- **一致性** → **内部自洽/与外部规范一致**
- **日志** → **可追溯性/决策依据记录**
